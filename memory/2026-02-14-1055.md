## 1) 這個 repo 是什麼？解決什麼問題？

它是一個 **Claude Code 的開發工作流插件（plugin）**，核心目標是把「AI 協作開發」變成一套可重複、可約束、可自動化的流程，而不是聊天式的隨機輸出。

它主打幾件事：

1. **90+ 工具/流程入口**（commands + skills + agents）
2. **最小化上下文占用（minimal context footprint）**
   - 把「永遠載入的規則」壓到很小（主要是 rules）
   - 其他能力用「按需載入」（skills / agents）
3. **強制工作流：Edit → Review → Gate**
   - 你只要改了檔案，就必須在同一輪把 Review/Precommit 跑完（可由 Stop hook 進一步強制）
4. **可選整合 OpenAI Codex（透過 MCP）**
   - 讓 Codex 當「第二雙眼睛」做獨立審查，而不是 Claude 自己自嗨

## 3) 這套插件的核心機制：Auto-Loop + Gate（最關鍵）

### (A) 行為層規則：`rules/auto-loop.md`
它把你（Claude）在改完檔後的「下一步」鎖死成一個 loop：

- 改 `.ts/.js/...` → **立刻跑 `/codex-review-fast`** → 通過後跑 `/precommit`
- 改 `.md` → **立刻跑 `/codex-review-doc`**
- Review 有問題 → 立刻修 → 立刻 re-review → 直到 ✅ Pass 才能停

重點是它寫得很硬：

- **Declaring ≠ Executing**：你說「我應該 review」不算，你要真的呼叫工具/命令
- **Summary ≠ Completion**：你寫一篇漂亮總結然後停下也算違規

### (B) hook 層強制（可選嚴格）：`hooks/stop-guard.sh`
Stop hook 會在你要結束對話/stop 時檢查：

- 你是否做過 code/doc 變更
- 是否已經跑過必要的 review/precommit
- review 是否最後結果是 pass（會掃 sentinel，或讀 state file）

它有兩種模式：

- `STOP_GUARD_MODE=warn`（預設）：只警告，但允許停
- `STOP_GUARD_MODE=strict`：**缺步驟就直接 block stop（exit 2）**

這代表：如果你在團隊裡想「強制 AI 不准半途收工」，開 strict 就能達到近似的效果。

## 4) Review 狀態追蹤：`.claude_review_state.json` 怎麼來的？

這套插件把「你改了什麼、review/commit 跑到哪」落地成狀態檔（repo root）：

### (A) PostToolUse：`hooks/post-edit-format.sh`
每次 `Edit/Write` 之後：

1. （可選）跑 prettier：若專案有 prettier config 或 node_modules bin
   - 可用 `HOOK_NO_FORMAT=1` 關掉
2. 依副檔名判定：
   - code 檔：`has_code_change=true`，並把 `code_review.passed=false`、`precommit.passed=false`（invalidate）
   - doc 檔：`has_doc_change=true`，invalidate `doc_review.passed`

### (B) PostToolUse：`hooks/post-tool-review-state.sh`
每次 Bash 或 MCP Codex 工具跑完後，它會解析輸出中的 sentinel 來更新：

- `code_review.executed/passed`
- `doc_review.executed/passed`
- `precommit.executed/passed`

它依賴這些標記（例如 `## Gate: ✅`、`✅ Ready`、`## Overall: ✅ PASS`）來做判定。

## 7) 兩個 runner 腳本（非常工程化的部分）

### `/precommit` runner：`scripts/precommit-runner.js`
特點：

- 自動偵測包管理器（yarn/pnpm/npm）
- 依 `package.json scripts` 決定要跑哪些步驟（沒有就 skip）
- 把完整 log 存到：`.claude/cache/precommit/<repoKey>/<shortSha>/`
- 會在 lint:fix 後抓 `git diff --name-only`，列出 auto-fix 改動的檔案

### `/verify` runner：`scripts/verify-runner.js`
特點：

- full 模式會嘗試：lint → typecheck → unit → integration（需指定檔案）→ e2e（需指定檔案）
- fast 模式主要跑 lint + unit
- 同樣有 cache/log 產出，並把結果用 Markdown 摘要輸出（方便進 Claude Code context）

### 為什麼要 runner？
因為 Claude Code 插件「無法可靠引用 plugin root 的 runner」的限制（README 也提到 upstream issue），所以它採用：

- **Try：如果目標專案根目錄有 runner（scripts/*.js）就跑**
- **Fallback：沒有 runner 就用偵測策略直接跑常見命令**

也就是：這 repo 同時提供「插件內 runner」與「落地到專案內 runner」兩條路。

## 8) 一個很實務的坑：插件命名空間（/sd0x-dev-flow:xxx）

Claude Code 安裝插件到外部專案後，指令會變成 namespaced：

- `/sd0x-dev-flow:codex-review-fast`

但插件內很多文件自然會寫短指令 `/codex-review-fast`。  
這 repo 在 `docs/features/plugin-namespace/...` 把問題講得很清楚：**CLAUDE.md 不會被插件自動載入**，所以不能靠 CLAUDE.md 來提醒「要加前綴」。

它的解法是：

- `hooks/hooks.json` 加了 **SessionStart hook**
- `scripts/namespace-hint.sh` 在 session 一開始輸出一句提示：
  > `all /command references should be invoked as /sd0x-dev-flow:command`

這招很聰明：用 hook 的 stdout 注入 Claude context，解決「文件內短命令引用」造成的執行失敗。

## 1) 先講結論：100% Codex-only 的可行性

- 呼叫 Codex
- 套用 Codex 產出（patch/apply）

### Level 1（最容易）：Codex-first，但仍由 Claude orchestrate（推薦）

- 讓 Codex 的輸出固定帶 sentinel（例如 `## Gate: ✅` / `✅ Ready` / `## Overall: ✅ PASS`），以便 `post-tool-review-state.sh` 正確更新狀態。
- “Codex 實作”要以 patch 形式回傳，避免它直接寫檔（更安全、更可控）。

### Level 2（中等）：把 sd0x-dev-flow 的 commands 全部變成 “Codex dispatcher”
做法：
- `/bug-fix`、`/feature-dev`、`/issue-analyze` 等命令不再自己做流程，而是：
  1) 收集必要上下文（git status/diff、檔案列表）

  3) 取得 structured output（plan + patch + commands to run + expected gates）
  4) 自動執行並回報

這樣插件就像 “Codex workflow adapter”。

- 會讓 skill 變成一堆「轉發/解析器」，維護成本反而上升。
- 一旦 Codex 輸出格式漂移，hook 的 sentinel/狀態解析容易壞。

### Level 3（最難）：離開 Claude Code plugin，把 hooks/rules 改成 Codex 自己的 runner/guard
做法：
- 不依賴 Claude Code hooks，而用：
  - git pre-commit hooks
  - 自己的 Node runner（類似 verify-runner/precommit-runner）
  - 或你自己的 agent framework
- Codex 變成 session host

這就已經不是 “改 repo 核心”，而是「把理念移植到 Codex-native」。
