好問題，**有關係，但不是同一層**。

- **Subagents（子代理）**：是「執行單位」  
  你可以把它當成一個獨立 worker（獨立 session）去做一件事，例如用 `sessions_spawn(...)` 丟一段任務給 `code-agent`。

- **team-tasks**：是「調度流程框架」  
  它用 JSON task/pipeline 管理整個流程（code → test → docs → monitor），決定先後、依賴、重試、結果記錄，再去派工給 agents（包含 subagents）。

所以：
- **Subagent = 工人**
- **team-tasks = 工地主任 + 排程板**

---

你現在的 team-task 流程其實已經明確寫到會用 subagent 來派工（Mode C 本地模式：`sessions_spawn(task=..., agentId=...)`）。  
Source: `memory/2026-02-16-0057.md#L127-L148`

如果你只是臨時叫一個 agent 幫忙，直接 subagent 就夠；  
如果你要可追蹤、可重跑、多人/多階段協作，team-tasks 比較適合。

## 我會怎麼做（用 trellis-2）

### 0) 先講現實限制（重點）
TRELLIS.2 很吃資源，不太適合直接跑在一般主機。  
我會把它當成 **GPU 推理服務**，前端只負責上傳、顯示進度、預覽與下載。

> 最低建議：Linux + NVIDIA GPU（最好 24GB VRAM 以上）。  
> 所以部署上多半是雲端 GPU 節點，不是純前端專案就能搞定。

---

## 1) 系統架構（我推薦）
**A. Frontend（Nuxt / Web）**
- 上傳圖片
- 顯示任務狀態（排隊 / 推理中 / 完成 / 失敗）
- 3D 即時預覽（Three.js / `<model-viewer>`）
- 下載 GLB / OBJ+貼圖

**B. API Backend（Node/FastAPI 都可）**
- 接收上傳、建立 job
- 佇列排程（Redis + worker）
- 查詢 job 狀態
- 產物檔案簽名下載 URL

**C. TRELLIS Worker（Python, GPU）**
- 吃 queue job
- 跑 trellis2 pipeline（image → mesh/material）
- 輸出 GLB
- 上傳到 Storage（S3/Supabase Storage）

**D. Storage + DB**
- Storage：原圖/輸出模型
- DB：job metadata、狀態、錯誤訊息、耗時

---

## 2) 「即時預覽」我會這樣定義
真即時（秒級）對 TRELLIS.2 不現實；我會做成**準即時體驗**：

1. 上傳後立刻進入 job 頁（有進度條）
2. 先回傳低模/草模（若可行）或 placeholder viewer
3. 完成後自動熱更新成最終 GLB
4. 一鍵下載模型

這樣使用者感受接近即時，但技術上可穩定。

---

## 3) API 設計（MVP）
- `POST /api/jobs`：上傳圖片並建立 job
- `GET /api/jobs/:id`：查狀態與結果 URL
- `GET /api/jobs/:id/stream`（SSE/WebSocket）：推送進度
- `GET /api/assets/:id/download`：下載模型

Job 狀態：
`queued -> preprocessing -> generating -> postprocessing -> done | failed`

---

## 4) 前端功能（你會看到的）
- 上傳區（拖拉/點選）
- 生成按鈕（可選品質：fast / balanced / high）
- 進度卡（百分比 + stage）
- 3D viewer（旋轉/縮放/背景/光源）
- 下載按鈕（GLB，之後可加 USDZ/OBJ）

---

## 5) 開發分期（我建議）
### Phase 1（MVP，先可用）
- 單張圖上傳
- 單 job 推理
- 完成後可預覽+下載 GLB

### Phase 2（體驗升級）
- 進度推送（SSE/WebSocket）
- 佇列管理與重試
- 歷史任務頁

### Phase 3（產品化）
- 使用者登入與配額
- 成本控管（每日上限、排隊優先級）
- 監控告警（GPU OOM、失敗率）

---

## 6) 風險與我會怎麼避
- **GPU 成本高** → 先做單 worker + 排隊，避免一開始過度配置  
- **推理慢** → 用 job 化 + 進度推送，不阻塞前端  
- **OOM** → 輸入圖尺寸限制 + 品質檔位 + 自動降級  
- **依賴難裝** → 嚴格用 Linux + CUDA 12.4 + conda，容器化固定環境
